# work — 작업 라우터 (앞단 디스패처)

입력을 받아 **버그냐 일반 태스크냐 분류**해서 알맞은 전문 스킬로 넘긴다.
너는 `/work` 하나만 기억하면 된다. 입력: `$ARGUMENTS`

```
/work <무엇이든>
   │ 분류
   ├─ 버그/오류/QA      → triage-fix  (원인 파악 중심)
   └─ 기능/개선/리팩토링 → task-fix    (설계 중심)
```

## 분류 원칙 (가장 중요)

**제목·키워드 하나로 단정하지 마라. 요구사항 전체를 읽고 종합 판단한다.**
- 입력을 **끝까지** 읽고, 실제로 **무엇을 해야 하는지(구현 항목)** 를 본다.
  팝업·버튼·링크 연결·"다시 보지 않음" 같은 기능·컴포넌트·동작·데이터 흐름이 하나라도 있으면
  그건 **코드 작업(`task-fix`)** 이다 — 제목이 "약관/정책/공지"라도.
- 표면 단어("약관" "디자인" "콘텐츠" 등)로 "코드 아님/범위 밖" 단정 **금지**. 그 단어 밑에
  실제 개발 항목이 깔려 있는지 본문을 확인한다.
- **혼합이면 나눠라.** 예: "약관 텍스트 개정(콘텐츠)" + "팝업 구현(기능)"이 섞이면,
  코드 부분만 떼어 task-fix로 가고, 비-코드 부분은 "이건 범위 밖(콘텐츠)"이라고 알린다.
- 진짜 코드 작업이 **하나도 없을 때만** 범위 밖으로 본다 (순수 문서 편집·운영·법무 텍스트).

## 분류 기준 (위 원칙 하에 참고 신호)

**→ 버그 (`triage-fix`)**:
- 기대와 다른 **현재 동작**을 고치는 일. "안 됨/오류/깨짐/~눌렀는데 안 감/잘못 표기" 류. QA 제보.

**→ 일반 태스크 (`task-fix`)**:
- **없던 걸 만들거나**(새 기능·컴포넌트·팝업·페이지) 기존을 **더 낫게** 바꾸는 일(개선·리팩·마이그레이션).
- "추가/만들/넣어/연결/개선/바꿔" 류 + 구현 항목 존재.

**→ 범위 밖** (어느 쪽도 아님):
- 코드 변경이 전혀 없는 순수 콘텐츠·문서·법무 텍스트·운영 작업. 이때만 멈추고 알린다.

## 동작

1. 입력이 링크면 소스(노션/슬랙)를 먼저 읽어 내용 파악(각 백엔드 스킬이 다시 읽으니 여기선 분류용으로만 가볍게).
2. 위 기준으로 분류.
3. **명확하면** 해당 스킬 절차를 그대로 수행한다 (triage-fix 또는 task-fix의 SKILL.md 흐름을 따른다).
   - 사용자에게 "버그로 보여 triage-fix로 진행할게요" 같이 **어디로 보냈는지 한 줄 알린다.**
4. **애매하면**(버그인지 개선인지 불분명, 둘 다 섞임) → `AskUserQuestion`으로 확인:
   > "이건 버그 수정인가요, 기능/개선 작업인가요?" (보기: 버그 / 기능·개선)
5. 분류 후엔 그 스킬의 가드·정지점·설정 로드(`triage.config.json`)를 **그대로** 따른다.

## 가드
- **키워드 하나로 단정 금지** — 본문 전체를 읽고 실제 구현 항목으로 판단(위 분류 원칙).
- **분류만 하고 끝내지 않는다** — 분류 후 해당 워크플로우를 끝까지(이슈→승인→PR) 진행한다.
- 애매하면 자동 단정 말고 **사용자 확인**.
- 분류 결과(어느 스킬로 갔는지)를 사용자에게 명시한다.
- 직접 `/triage-fix`·`/task-fix`를 호출해도 되는 사용자에겐, `/work`는 "고민 없이 던지는" 단일 입구일 뿐임을 안다.
